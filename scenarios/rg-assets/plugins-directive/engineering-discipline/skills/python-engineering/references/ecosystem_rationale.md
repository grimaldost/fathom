# Python Ecosystem & Stack Rationale

This file explains *why* we recommend specific tools and patterns. Read this
when the user asks for justification or when comparing alternatives.

## 1. The Ecosystem-First Principle

Before implementing custom logic, evaluate existing solutions within the Python
ecosystem. Custom build scripts are liabilities; standardized tools are assets.

## 2. Package & Dependency Management — `uv`

`uv` is a high-performance replacement for pip, virtualenv, poetry, pyenv, and
pip-tools — all in a single binary written in Rust.

- **Speed**: 10-100× faster than pip. Parallel downloads, efficient caching
  with hard-links, and Rust-native dependency resolution.
- **Determinism**: Cross-platform `uv.lock` file pins every transitive
  dependency. Use `uv sync --frozen` in CI for byte-identical installs.
- **Python management**: `uv python install` / `uv python pin` replaces
  pyenv. No need for a separate Python version manager.
- **PEP 735 support**: `[dependency-groups]` for dev deps that never ship.
- **Adoption**: Majority CI share in major projects. Poetry and PDM are
  declining. Rye was archived in February 2026, fully superseded by uv.

## 3. Repository Layout — The `src` Standard

The `src` layout (`src/my_package/`) is the standard for any packaged or
production project.

- The legacy "flat" layout allows the "implicit import" problem, where tests
  run against local source files rather than the installed package.
- The `src` layout forces tests to run against `site-packages`, ensuring
  that what you test is exactly what you ship.
- This applies to apps as well as libraries; the one real exception is a
  throwaway single-file `uv run` script.

## 4. Build Backend — `uv_build`

`uv_build` is the default backend generated by `uv init --lib`. It is PEP
621-compliant and integrates natively with the uv toolchain.

- **Why not Hatchling?** Hatchling is still excellent and fully supported.
  `uv_build` is preferred for new projects because it's generated by default
  and reduces the dependency footprint. For existing projects on hatchling,
  there is no urgency to migrate.

## 5. Linting & Formatting — `ruff`

`ruff` replaces Flake8 (plus dozens of plugins), Black, isort, pydocstyle,
pyupgrade, and autoflake — all in a single Rust binary.

- **Speed**: 10-100× faster than any individual tool. Sub-second on large
  codebases, practical as a commit hook.
- **Unified config**: One `[tool.ruff]` section replaces five tool configs.
- **500+ rules**: Compatible with Flake8 rule ecosystem.
- **2026 style guide**: Ruff v0.15+ introduced the 2026 formatter style,
  including improved lambda formatting and PEP 758 unparenthesized except.

## 6. Type Checking — `ty` (primary) / `mypy` (stable)

### ty (Astral, beta)

`ty` is Astral's Rust-based type checker and language server, released in
beta in December 2025. It is part of the same Astral toolchain as uv and ruff.

- **Speed**: 10-100× faster than mypy on large codebases (Astral-reported), with very fast incremental re-checks.
- **Language server**: Full LSP with go-to-definition, auto-import, rename,
  inlay hints — replaces Pylance/Pyright for editor integration.
- **Advanced type system**: First-class intersection types, advanced type
  narrowing, sophisticated reachability analysis.
- **Gradual typing friendly**: Avoids false positives on untyped code.

### mypy (stable)

mypy is the battle-tested standard with the broadest plugin ecosystem
(django-stubs, sqlalchemy-stubs, pydantic plugin, etc.).

- Use mypy when you need plugin support that ty doesn't yet provide.
- Use mypy as a secondary CI check alongside ty for extra safety.
- mypy strict mode remains the gold standard for fully-typed codebases.

### Pyright (VS Code)

Pyright powers Pylance in VS Code. With ty's language server now available,
Pyright is no longer the default recommendation for new setups, but it
remains a valid choice — especially for teams already invested in it.

## 7. Typing Philosophy — Protocol over ABC

**`typing.Protocol`** (PEP 544) is the recommended way to define interfaces
in modern Python. It enables structural subtyping (static duck typing):
any class that implements the required methods satisfies the Protocol,
without needing to inherit from it.

This matters because:

- **Decoupling**: Concrete classes don't need to know about the Protocol.
  You can define a Protocol for third-party classes you don't control.
- **Testability**: Mock any dependency by implementing the Protocol's methods.
  No need for `unittest.mock.patch` or complex test fixtures.
- **Pythonic**: Protocols align with Python's duck-typing philosophy. If it
  walks like a duck and quacks like a duck, the type checker agrees.

**When to use ABCs instead**: ABCs remain valid when you need shared method
implementations via inheritance, runtime `isinstance()` checks without
`@runtime_checkable`, or `@abstractmethod` enforcement that prevents
instantiation of incomplete subclasses.

**When to use neither**: For simple data shapes, use `dataclass` or
Pydantic `BaseModel` — not Protocols or ABCs.

## 8. Architecture — Functional Core / Imperative Shell

The recommended application architecture separates pure business logic from
I/O and framework code:

- **Functional core** (`core/`): Pure functions and data classes. No I/O,
  no framework imports, no side effects. Fully testable with unit tests.
- **Imperative shell** (`interface/`): Thin adapters that handle I/O —
  FastAPI routes, CLI commands, database clients, HTTP clients. These
  orchestrate the core and translate between external formats and domain
  objects.

The layers connect via **Protocol interfaces** and **constructor injection**.
A service in `core/` declares its dependencies as Protocol-typed constructor
parameters. The `interface/` layer provides concrete implementations.

```python
# core/services.py
from typing import Protocol

class PaymentGateway(Protocol):
    async def charge(self, amount: Decimal, currency: str) -> str: ...

class PaymentService:
    def __init__(self, gateway: PaymentGateway) -> None:
        self._gateway = gateway

    async def process(self, order: Order) -> Receipt:
        txn_id = await self._gateway.charge(order.total, order.currency)
        return Receipt(order_id=order.id, transaction_id=txn_id)
```

This is essentially **hexagonal architecture** (ports and adapters) without
the ceremony. The core defines ports (Protocols), and the interface provides
adapters (concrete implementations).

### Dependency Injection

**Constructor injection** is the Python norm. Pass dependencies as parameters
to `__init__`. For most projects, this is sufficient — no DI framework needed.

For **FastAPI** projects, use `Depends()` for request-scoped dependencies.
This is FastAPI's native DI system and handles the most common case elegantly.

For **complex dependency graphs** (10+ services with shared state), consider
`dependency-injector` (~4,800 GitHub stars). But reach for it only when
constructor injection becomes unwieldy.

## 9. Configuration Management — `pydantic-settings`

Never read `os.getenv()` directly. `pydantic-settings` provides:

- **Typed config**: Define a strict schema for environment variables.
- **Fail-fast**: Missing required config raises at startup, not at runtime.
- **Security**: `SecretStr` masks sensitive values in logs and repr.
- **Validation**: Pydantic's full validation engine for config values.

## 10. Observability — `structlog` + `opentelemetry`

Modern observability has three pillars: **logs**, **traces**, and **metrics**.

**structlog** handles structured logging: JSON for machines in production,
pretty-printed for humans in development. Combined with `contextvars` for
request-scoped data and `stdlib.ProcessorFormatter` for bridging third-party
loggers.

**OpenTelemetry** handles distributed traces and metrics. The Python SDK's
traces and metrics signals are both stable. The consensus integration pattern
uses a custom structlog processor to inject `trace_id` / `span_id` from
the active OTel span into every log entry, enabling correlated debugging.

See `references/observability.md` for the full setup.

## 11. Async-First for I/O

Default to `async`/`await` for web services and I/O-bound applications.
The async ecosystem has matured to the point where there's no reason to
default to synchronous code for new projects.

Key ecosystem choices:

- **HTTPX**: The modern default HTTP client. Sync and async APIs, HTTP/2
  support, Requests-compatible interface. Use for all new code.
- **asyncio.TaskGroup** (3.11+): Replaces `asyncio.gather()` for concurrent
  operations. Provides automatic cancellation on failure and proper
  exception handling through ExceptionGroups.
- **Uvicorn**: The safe default ASGI server. Granian (Rust-based) is an
  emerging alternative with significantly better throughput. The
  Gunicorn + Uvicorn workers pattern is legacy — Uvicorn has native
  worker management since 0.30.0+.

## 12. Application Interfaces

We decouple "Core Logic" from how the user interacts with it.

- **CLI**: `typer` — type-hint-driven argument parsing, auto-generated help.
- **Web**: `fastapi` — Pydantic integration, async-native. Treat as an
  *interface layer*; keep business logic in `core/`.

## 13. Testing — `pytest` + `hypothesis`

Simple `assert` syntax, powerful fixtures, and ecosystem dominance. Hypothesis
adds property-based testing for data-heavy code, catching edge cases that
manual parameterized tests miss. See `references/testing_and_qa.md` for full
patterns including snapshot testing and testcontainers.

## 14. Python 3.14 Features Worth Adopting

- **PEP 649 (deferred annotations)**: Drop `from __future__ import annotations`
  on 3.14+. Forward references work natively.
- **PEP 750 (t-strings)**: Template strings for safe SQL, HTML, logging.
  Ecosystem still nascent — adopt when useful.
- **PEP 758 (unparenthesized except)**: Minor syntax sugar.
- **PEP 784 (zstandard)**: `compression.zstd` in stdlib — better than gzip.
- **PEP 734 (subinterpreters)**: Watch but don't adopt yet. API is new.
- **Free-threaded Python**: Supported in 3.14 but requires separate build.
  Production-ready for specific workloads, not a general recommendation yet.
